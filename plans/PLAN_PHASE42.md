## Context

Six prior phases ship the OIDC mTLS introspection surface:

| Phase | What | Public API |
|-------|------|------------|
| 24 (v0.25.0) | RFC 8705 `tls_client_auth` | `with_introspection_mtls`, `..._combined` |
| 25 (v0.26.0) | RFC 8705 `self_signed_tls_client_auth` | `with_introspection_self_signed_mtls`, `..._combined` |
| 33 (v0.34.0) | Atomic cert/key rotation | `reload_mtls_identity`, `..._combined` |
| 35 (v0.36.0) | Filesystem-watcher auto-reload | `watch_mtls_files` |
| 37 (v0.38.0) | Trait-based `MtlsIdentityProvider` | `watch_mtls_provider` |
| 39 (v0.40.0) | Auto refresh-on-handshake-failure | `with_mtls_refresh_hook` |

Every test today is **builder-level**: it asserts that PEM
parsing succeeds, that the auth-method enum flips correctly,
that `mtls_client.is_some()`, etc. None of them performs an
actual mTLS handshake against a server that requires client
cert auth. The PLAN.md backlog explicitly calls this out:

> **OIDC mTLS end-to-end integration test** — Phases 24 + 25
> ship builder-level tests; a real TLS server requiring client
> auth (axum-server + rustls + per-test CA) would close the
> loop. ~300 lines of test infra.

Phase 42 closes the loop.

## Why now

- The mTLS surface is feature-complete after Phase 39 (handshake-
  failure refresh) + Phase 40 (Python facade for the refresh
  hook). No further mTLS work is queued, so a defensive test
  pass is appropriate before unrelated work touches this code.
- The Phase 41 security thread (jsonwebtoken bump) is the
  natural lead-in to a security-focused integration test.
- The `rcgen` + rustls + per-test-CA pattern is already
  proven in [`crates/tako-mcp/tests/grpc.rs`](../crates/tako-mcp/tests/grpc.rs#L200)
  (Phase 5.B mTLS coverage for the gRPC MCP transport). Phase
  42 lifts the same pattern into `tako-compat` for the OIDC
  introspection path.

## What this phase will land

### 42.A — `with_introspection_mtls_extra_root` builders

Today's `with_introspection_mtls` and
`with_introspection_self_signed_mtls` both build a
`reqwest::Client` like this:

```rust
let client = reqwest::Client::builder()
    .timeout(Duration::from_secs(10))
    .identity(identity)
    .build()?;
```

The Client trusts only the system + webpki-roots CA store. That
works against public-CA-signed OIDC issuers (Auth0, Okta,
etc.) but **not** against enterprise private OIDC deployments
behind a private CA — a common case at the operator profile
this surface targets (mTLS introspection is itself a
corporate-feature). Today operators have to drop down to a
custom `reqwest::Client` and skip the tako builder, losing
the rotation / hook plumbing.

Phase 42 adds two new builders:

```rust
pub fn with_introspection_mtls_extra_root(
    self,
    cert_pem: &[u8],
    key_pem: &[u8],
    extra_root_ca_pem: &[u8],
) -> Result<Self, TakoError>;

pub fn with_introspection_self_signed_mtls_extra_root(
    self,
    cert_pem: &[u8],
    key_pem: &[u8],
    extra_root_ca_pem: &[u8],
) -> Result<Self, TakoError>;
```

Both delegate to a shared internal helper that wraps the
existing `reqwest::Client::builder().identity(...).build()` with
one extra `.add_root_certificate(...)` call per cert in
`extra_root_ca_pem`. The PEM blob accepts a single root or a
concatenated bundle (multi-cert PEM); behavior matches
[`Self::with_introspection_mtls`] otherwise (silent no-op when
`introspection.is_none()`, identity errors surface as
`TakoError::Invalid`).

**Why both an `extra_root` AND the existing builders?** The
default builders stay zero-CA-config for the common
(public-CA) deployment shape. Operators behind a private CA
opt into the new builder.

The `_combined` PEM siblings are out-of-scope; operators can
pass the same PEM-bundle byte slice for cert / key when
their PKI happens to combine them.

### 42.B — Integration test

New file `crates/tako-compat/tests/oidc_mtls_e2e.rs` (gated to
`feature = "oidc"`).

Per-test fixture generation (single rcgen pass at test
setup):

| Cert | Signed by | Used by |
|------|-----------|---------|
| `test-ca` (root) | self | Client trust + server-side `WantClientCert` verifier |
| `localhost` server leaf | `test-ca` | axum-server rustls `ServerConfig` |
| `oidc-client` client leaf | `test-ca` | tako-compat `Identity` (presented in handshake) |
| RS256 JWT signing keypair (rcgen-independent — direct via `jsonwebtoken`) | self | JWKS publication + test JWT signing |

Servers spun up:

1. **Plain HTTP wiremock** (already in dev-deps) for:
   - `GET /.well-known/openid-configuration` — minimal RFC 8414
     discovery doc with `issuer`, `jwks_uri`, `introspection_endpoint`
     pointing at the mTLS server.
   - `GET /jwks` — publishes the RS256 JWT public key as a JWK.
2. **HTTPS `axum-server`** (rustls + `RequireAndVerifyClientCert`)
   for:
   - `POST /introspect` — accepts URL-encoded form body, asserts
     `token=` + `client_id=` are present, returns
     `{"active": true, "sub": "alice", "aud": "test-audience"}`.

Resolver wiring:

```rust
let r = OidcAuthResolver::discover(&plain_http_issuer, "test-audience").await?
    .with_introspection_uri(&https_introspect_uri, "client", None)
    .with_introspection_mtls_extra_root(
        client_cert_pem,
        client_key_pem,
        ca_pem,
    )?;
let principal = r.resolve(&signed_jwt).await?;
```

Tests:

| Test | Assertion |
|------|-----------|
| `mtls_round_trip_with_client_cert` | Full happy-path: discovery → JWKS → JWT validation → mTLS POST `/introspect` → `active=true` → `Principal` returned with the right tenant/user. |
| `mtls_rejects_handshake_without_client_cert` | Resolver built via the **non-mTLS** `with_introspection` path → POST hits the server → server's `WantClientCert` policy aborts the handshake → `TakoError::Transport`. |
| `mtls_rejects_handshake_with_wrong_ca` | Client cert is signed by an unrelated CA (rcgen second-pass) → server rejects → Transport error. |
| `self_signed_mtls_round_trip_with_extra_root` | Same happy-path but using the `_self_signed_` sibling builder + `RFC 8705 §2.2` `self_signed_tls_client_auth` auth method. |
| `extra_root_unparseable_pem_errors_at_builder_time` | Pass garbage bytes for `extra_root_ca_pem` → builder returns `TakoError::Invalid`, not a runtime handshake failure. |

The existing `tako-mcp/tests/grpc.rs` `ensure_crypto_provider`
once-pattern is reused so multiple test binaries can install
`aws_lc_rs` as the rustls default provider exactly once.

## Out of scope

- **`_combined` PEM sibling for the new builders.** Operators
  with combined cert+key PEMs can pass the same byte slice
  twice; a dedicated `_combined` helper is sugar that the
  existing `with_introspection_mtls_combined` pattern doesn't
  obviously generalize across an additional CA argument.
  Defer until ask.
- **Custom CA support for non-introspection endpoints (JWKS,
  discovery).** The default `OidcAuthResolver` HTTP client is
  built inside `discover()` with no public injection point.
  Adding a `with_extra_root_ca` builder for the resolver-wide
  client is a larger ask (touches discovery boot path,
  `discover()` signature, `Client` lifecycle). Defer until an
  operator asks.
- **TLS 1.2 fallback / cipher-suite control.** rustls 0.23
  defaults to TLS 1.3-only; the current default is fine.
- **gRPC mTLS introspection.** RFC 8705 is HTTP-only; gRPC
  introspection isn't a thing yet.
- **Strict-allowlist mode** for the URL pre-fetch surface
  (URL-source images). Unrelated to mTLS introspection;
  separate roadmap item.
